# Iron Throne of Ashes — Analysis & Ground-Up Redesign Roadmap

> Status: Analysis (no code changed)
> Date: 2026-06-21
> Author: Claude (codebase survey)
> Companion to: `docs/prd.md`, `docs/architecture.md`

This document answers three questions:
1. **What is this game trying to be?** (design intent)
2. **What does it actually do today, and why does it feel broken?** (mechanics, turn structure, AI, root-cause of the play-test pain)
3. **How do we get to a clean ground-up redesign?** (roadmap)

---

## Part 1 — What the game is trying to be

A **2–4 player digital board game** about *dynastic rivalry under a shared apocalypse*. Players are
Arch-Regents competing to control territory (Strongholds) on a fixed 28-node map, **while jointly
holding back an autonomous AI antagonist** (the Shadowking / "Wraith Tyrant") whose advance is tracked
by a shared **Doom Toll**. If the Toll maxes out, *everyone* loses.

The intended emotional engine is **negotiation under pressure**: each round the table must collectively
pay (in Fate Cards) to block the antagonist, but the player in the lead has the most to lose, so trailing
players can refuse to pay as "political taxation." Three modes (Competitive, Cooperative, Blood Pact /
traitor) reuse one board and ruleset; only win/loss conditions change. **No elimination** — a beaten
player enters *Broken Court* (reduced actions, full voting rights) and can be *Rescued*.

The whole thing runs on the **Alliance Engine**: every in-world noun is a swappable **GLL token**, so the
same engine can be reskinned (e.g. "Sea of Knives") with no logic changes.

**The design is genuinely good and the PRD is unusually complete.** The problem is not the design — it is
the implementation's *drift away from* the design and a UI/onboarding layer that never made the rules
legible to a human at the keyboard.

## Part 2 — Mechanics & turn structure (as implemented)

### Round structure (4 phases, fixed order)
`shadowking → voting → action → cleanup`, repeated each round. Player turn order within the Action Phase
is a fixed seeded shuffle for the whole game; the Voting Phase is simultaneous.

1. **Shadowking Phase** — draw one Behavior Card (two in Final Phase). The card is *revealed but not yet
   resolved*.
2. **Voting Phase** — every active Arch-Regent simultaneously votes `COUNTER` or `ABSTAIN`. Unanimous
   COUNTER (with each player paying the Fate-Card cost) blocks/reduces the card; any abstention (or anyone
   who can't pay) lets it resolve **and advances the Doom Toll +1**. *Then* the card resolves using the
   vote outcome.
3. **Action Phase** — each player spends actions (2 normal, 1 if Broken): move (1 Banner/node), claim a
   Stronghold (1 Banner), recruit (Herald only), initiate combat, rescue, Herald diplomacy.
4. **Cleanup Phase** — discard unspent Banners, regenerate Banners, replenish Fate hands, check Territory
   Victory, advance round.

### Subsystems
- **War Banners** — the one resource. Generated by Artificers (×3 at a Forge Keep). Do not persist
  between rounds. Pay for movement, claims, and add to combat strength.
- **Fate Cards** — personal hand (limit scales with Heralds). Spent to vote and to rescue; drawn
  temporarily for combat. Deck reshuffle advances the Doom Toll.
- **Combat (War Field)** — base strength = sum of character Power Levels + Banners; both sides add a
  drawn Fate Card; higher total wins, ties to defender; loser takes Penalty Cards equal to the margin.
- **Broken Court** — when Penalty Cards ≥ Banners, the player is reduced (1 action, no claim/combat/
  recruit) but keeps full voting rights and can be Rescued (donate Fate Cards → clears penalties).
- **Doom Toll** — shared track. Advances on failed votes, deck reshuffles, Forge captures, Broken-Court
  entry; recedes on Death Knight kills, Forge reclaims, unanimous votes. Crossing the Final-Phase
  threshold triggers escalation (2 cards/round, blight auto-spread, double vote cost).
- **Victory** — Territory (hold the Heartstone + most Strongholds, checked at Cleanup); Doom Complete
  (everyone loses, or the Blood Pact traitor wins); All-Broken Draw.

### How the computer-driven characters work
- **The Shadowking is not a player** — it is a deterministic card executor. Each round it draws a Behavior
  Card and resolves the mapped effect (`spawn`/`move`/`claim`/`assault`/`escalate`). Its forces are
  Death Knights (mobile, power 10) and Blight Wraiths (static spreaders, power 6). All logic in
  `src/systems/shadowking.ts`, deterministic from the seed.
- **AI players** fill empty seats at three tiers (`src/systems/ai-player.ts`):
  - *Apprentice* — nearest unclaimed Stronghold, almost always COUNTERs.
  - *Knight-Commander* — prefers Forge Keeps, takes opportunistic combat when ≥20% stronger, abstains ~12%.
  - *Arch-Regent* — full heuristic: forge targeting with "don't help the leader," threat avoidance,
    rescue missions, leader-check voting (abstains when the card targets a rival leader).
  All AI is seeded/deterministic.

## Part 3 — Why it feels broken (root-cause of the play-test report)

Every complaint from the play-test maps to a specific, identifiable cause:

### 3.1 "The intro screen about voting is gibberish"
`src/ui/social-pressure-onboarding.ts` shows the PRD's *Social Pressure* text **before the first game**:
an abstract essay about "political taxation on the frontrunner" and "Forge Keep dominance" — concepts a
brand-new player has zero referents for. It is shown *before* they have seen a board, a Stronghold, or a
Doom Toll. It is pedagogically backwards: political theory before any concrete noun exists. It also still
says "Final Phase (Doom Toll ≥ 10)" while the live threshold is **8** (see 3.5).

### 3.2 "There is no onboarding tutorial"
**The tutorial is fully built but unreachable.** `tutorial-script.ts`, `tutorial-state.ts`, `tutorial.ts`
and the controller's tutorial branch all exist — but `src/ui/mode-select.ts` only offers Competitive /
Blood Pact / Cooperative. `'tutorial'` appears only as a *type union member*; there is no button that
selects it. `index.ts` even comments "cast includes 'tutorial' for future ModeSelectUI expansion." So a
first-time player is dropped straight into a full competitive game with no teaching path.

### 3.3 "The board seems to have an idea but is nonsensical"
`board-renderer.ts` draws a hand-placed **radial/polar diagram** (keeps on an outer ring, forges mid-ring,
Dark Fortress center) of 28 dots with lore names ("Shadow Gate East", "Fire-Wind Bridge") and no legend,
no "this is YOU" emphasis, and player tokens drawn *below* nodes where they collide with labels. There is
no camera framing your court, no indication of what is claimable/reachable until mid-action. It reads as
an abstract circle of cryptic dots.

### 3.4 "Clicking advances turns with no meaningful feedback"
`GameController` drives the entire game as an `async` `while` loop (`runGame → runRound → run*Phase`) that
`await`s a promise resolved by clicks. Failure modes a player hits:
- `handleNodeClick` **silently returns** on most clicks (not adjacent / can't afford / wrong phase) — no
  message, no flash. Clicking "around the map" mostly does nothing, occasionally consumes an action.
- The action HUD (`updateHUD`) is a raw `innerHTML` string with **no stylesheet linked** in `index.html`
  (there is no `hud.css`), so the "Actions / Banners / End Turn" affordance is unstyled and easy to miss.
- After your two actions silently resolve, control passes to AI opponents that act on 750 ms timers, so
  from the player's seat: rounds advance, things move, and nothing explains why.
- An `EventFeed` was recently added (untracked `event-feed.ts`/`.css`) and is the *only* real feedback
  channel — but it is a side log, not the primary loop.

### 3.5 The deeper rot: the code has drifted from its own spec, and the test suite is red
The `0acbe6c "initial ML testing"` commit re-tuned balance constants but **did not reconcile the spec,
the UI strings, or the tests**:

| Value | PRD / CLAUDE.md | Live code (`game-state.ts`) | UI strings still say |
|---|---|---|---|
| Behavior deck | 6/6/4/3/1 (spawn/move/claim/assault/escalate) | **5/4/7/4/0** (no ESCALATE at all) | — |
| `DOOM_TOLL_MAX` | 13 | **11** | "reached 13" (game-over screen) |
| Final-Phase threshold | 10 | **8** | "reached 10" / "≥ 10" (onboarding, tooltip) |
| Herald diplomacy relief | 2 | **3** | — |
| Banner production | per-Artificer | changed (tests now expect different) | — |

Consequences:
- **50 tests fail across 9 suites** (`npm test`): game-state, behavior-deck, doom-toll, resources, voting,
  shadowking, game-modes, game-loop, and the controller E2E. Typecheck is clean, but the suite is red.
- The **pre-push hook runs lint + test**, so per the project's own rules the repo is currently
  **un-pushable** and in a "broken state" that must be fixed before anything ships.
- There is **no single source of truth** for balance: constants live in `game-state.ts`, are *described*
  in the PRD and CLAUDE.md, are *hardcoded as English* in UI strings, and are *re-tuned* by the ML harness
  — four copies, none agreeing.

### 3.6 Architectural spaghetti (the structural cause of the above)
The *engine* (`models/`, `systems/`, `engine/`) is mostly clean, pure, and seeded — it is salvageable.
The rot is concentrated in the **UI/controller seam**:
- `GameController` (1,196 lines) is simultaneously the **simulation driver**, the **UI orchestrator**, the
  **AI runner**, the **tutorial engine**, and the **multiplayer client**. One class, five jobs.
- **Core rules are reimplemented 3–4 times.** "Move one node + spend a banner + maybe claim + maybe fight"
  exists separately in `handleNodeClick`, `autoResolveActions`, `runAITurn`, and `runScriptedOpponent`
  (and again in `simulation.ts` and `multiplayer`). Grep shows `fellowship.currentNode =` mutated in 10
  places and `strongholdsClaimed += 1` in 7. These will drift out of sync — and already have.
- **UI mutates state directly** instead of issuing intents through the systems layer, so the systems'
  validation/logging is bypassed on the human path.
- **Rendering is imperative and ad hoc** — `renderAll()` pokes a few components; most state changes don't
  re-render the HUD; there is no single "state → view" function.

## Part 4 — Ground-up redesign roadmap

**Guiding decision: rebuild the UI and the control flow from scratch; *consolidate and keep* the engine.**
The models/systems are well-factored and well-specified — throwing them away would discard the one solid
asset. The redesign is therefore "new front-half, hardened back-half."

### Phase 0 — Establish ground truth (prerequisite, ~days)
*You cannot redesign on top of a red suite and four disagreeing copies of the rules.*
1. **Decide the canonical balance numbers.** Either accept the ML-tuned values (and rewrite the PRD +
   CLAUDE.md + tests to match) or revert to the PRD values. One decision, applied everywhere.
2. **One source of truth for constants.** All balance lives in `game-state.ts`; PRD references it; UI
   reads it (never hardcodes "13"/"10"); the ML harness writes back to it via a documented process.
3. **Get the suite green.** Reconcile the 50 failing tests against the canonical numbers. After this, the
   pre-push hook passes and the repo is shippable-state again.
4. **Write down the determinism contract** (seed → identical game) as an explicit, tested invariant.

### Phase 1 — Consolidate the engine into one command/reducer API (~1–2 weeks)
1. Collapse the 3–4 duplicate movement/claim/combat implementations into a single
   `applyCommand(state, command) → { state, events[] }` reducer. Every actor — human, AI, scripted
   tutorial, simulation, multiplayer — goes through it. No direct state mutation anywhere else.
2. Define a typed **command** vocabulary (`MOVE`, `CLAIM`, `RECRUIT`, `RESCUE`, `INITIATE_COMBAT`,
   `DIPLOMACY`, `VOTE`, `END_TURN`, `ACCUSE`) and a typed **event** stream (the existing `actionLog`,
   formalized) that is the *only* thing the UI consumes.
3. Make the phase sequencer a pure state machine separate from any timing/animation/`await`.

### Phase 2 — Rebuild the UI as render-from-state (~3–4 weeks; the bulk of the work)
1. **Unidirectional data flow.** `state → render(state)`; user gestures emit commands; the reducer
   produces the next state; re-render. Pick the lightest tool that enforces this (a tiny store +
   render function is enough; a small framework is fine). Kill the imperative `renderAll()`-pokes-things
   pattern and the `await`-a-click game loop.
2. **A board a human can read.** Reconsider the radial layout. Frame the player's own court; show, at all
   times and without hovering: who owns what, where *you* are, what is reachable (Banner-affordable) and
   claimable this action, and where the antagonist forces are. Tokens must not collide with labels.
3. **The persistent HUD the PRD already specifies (F-016)**, actually built and styled: Doom Toll + phase,
   every player's Stronghold/Banner counts, turn-order tracker, Fate-deck depth, "whose turn." This is the
   "know who's winning at a glance" requirement — currently unmet.
4. **Feedback on every action and every rejection.** Promote the EventFeed to a first-class element.
   Illegal clicks must explain themselves ("not adjacent", "need 1 Banner", "not your turn"). Each
   resolved action animates/logs. The Shadowking's move is shown and narrated before player turns.

### Phase 3 — Onboarding that actually onboards (~1–2 weeks)
1. **Wire the tutorial into mode select** (it already exists — just unreachable). Make a fresh first run
   default to it.
2. **Replace the abstract Social-Pressure wall** with *in-context* teaching: introduce one concept at the
   moment it first matters (per PRD F-012's per-turn plan), not as an essay up front. Keep the political-
   theory framing as an *optional* "About the Voting Phase" entry in Settings, not a gate.
3. Verify PRD's "Tutorial Turn 3 (first combat) must not silently fail" with a real click-through.

### Phase 4 — Re-balance against the locked engine (~1 week + sim time)
1. Re-run the UGT/ML harness against the Phase-0 canonical constants.
2. Confirm the PRD targets (Dark Lord win rate 18–22%, 8–16 rounds, etc.) and write results back to the
   single source of truth, updating PRD + tests in lockstep so drift can't recur.

### Cross-cutting principles for the rebuild
- **No rule logic in the UI.** The UI renders state and emits commands. Period.
- **No balance number outside `game-state.ts`.** UI reads constants; never prints literals.
- **Green tests are the definition of done** for every phase (project rule: checks are sacred).
- **Keep the GLL + SeededRandom contracts** — they are the engine's two best ideas.

### What to keep vs. rebuild (summary)
| Layer | Verdict |
|---|---|
| `gll/`, `utils/seeded-random`, `models/` | **Keep** — clean, the engine's foundation |
| `systems/` | **Keep & consolidate** — sound rules, but de-duplicate the action logic into one reducer |
| `engine/game-loop`, `simulation` | **Refactor** — extract a pure phase machine; reuse for sim + play |
| `ui/game-controller.ts` | **Rebuild** — split driver / orchestrator / AI-runner / multiplayer |
| `ui/board-renderer.ts` | **Rebuild** — readable layout, render-from-state, legible affordances |
| `ui/mode-select`, onboarding, tutorial wiring | **Rebuild** — reachable tutorial, in-context teaching |
| Balance constants / PRD / tests | **Reconcile to one source of truth (Phase 0)** |

## Appendix — Fastest path to "playable enough to evaluate the design"
If the goal is to *feel the design* before committing to the full rebuild, the minimum is:
1. Phase 0 (green tests + canonical numbers).
2. Add a **Tutorial** button to mode select (≈1 line of UI + existing code).
3. Style the HUD and make illegal clicks explain themselves (small, high-leverage).
That alone converts "unintelligible" into "a rough but legible game," and de-risks the larger rebuild.
